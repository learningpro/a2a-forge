# Project Research Summary

**Project:** A2A Workbench
**Domain:** Protocol-specific AI agent testing desktop application (Tauri 2.x + React + Rust)
**Researched:** 2026-03-24
**Confidence:** HIGH

## Executive Summary

A2A Workbench is a purpose-built desktop tool for developers testing agents that implement the A2A (Agent-to-Agent) protocol. The product occupies a gap between general-purpose API tools (Postman, Insomnia) and the A2A-specific developer workflow: those tools show raw HTTP, but A2A agents expose skills, produce structured task lifecycles, and stream results via SSE — none of which generic tools surface meaningfully. The recommended approach is a Tauri 2.x desktop app with a React/TypeScript frontend and Rust backend, where all network traffic (HTTP, SSE) runs through Rust, not the webview. This is not a stylistic choice — it is a hard requirement: webview fetch calls hit CORS errors against arbitrary agent URLs, and SSE from the browser's `EventSource` API fails at the CORS layer before a single event arrives.

The recommended stack is well-settled: Tauri 2.10.3, React 18, TypeScript 5, Vite 6, Tailwind CSS 4, Zustand 5 for UI state, TanStack Query 5 for server state, Monaco Editor for JSON input, SQLite via tauri-plugin-sql for structured persistence, and tauri-plugin-keyring for credential storage. The architecture follows a clean three-panel layout (agents, skills, test) backed by four domain-scoped Zustand stores and a typed Rust command boundary generated by tauri-specta. SSE streaming uses Tauri Channels (not global events) for ordered, per-task delivery.

The two highest-risk areas are: (1) the Tauri v2 permissions ACL system — every plugin starts disabled and misconfiguration produces silent failures that are routinely mistaken for code bugs; and (2) the A2A protocol's immaturity — the spec has already changed since its April 2025 launch and will continue evolving, so all parsers must treat non-core fields as optional. Both risks must be addressed in Phase 1, not deferred. Code signing for macOS distribution is the one item that can safely wait until a distribution milestone, but it requires a paid Apple Developer account and should be budgeted from day one.

## Key Findings

### Recommended Stack

The Tauri 2.x + React + Rust combination is the correct choice for this tool. Electron would work but produces binaries 10x larger and forces SSE through the webview where CORS is a constraint. The Rust backend handles all outbound network (reqwest + reqwest-eventsource for SSE), all database access (sqlx via tauri-plugin-sql), and all credential operations (tauri-plugin-keyring). The React frontend handles only display and user interaction, communicating with Rust exclusively through tauri-specta generated typed bindings — raw `invoke()` string calls should not appear in production code.

One dependency is flagged as unstable: `tauri-plugin-keyring` v0.1.0. It is the only cross-platform OS keychain option for Tauri 2, but it is community-maintained and marked unstable. A contingency plan (encrypt credentials in SQLite using a machine-derived key) should be documented in Phase 1 in case it proves unreliable.

**Core technologies:**
- Tauri 2.10.3: Desktop shell and Rust backend — smaller binary, native sandbox, CORS-free network
- React 18 + TypeScript 5: UI layer — required by TanStack Query v5 and Zustand v5
- Vite 6: Bundler — official Tauri recommendation, required by Tailwind CSS 4 plugin
- Rust (stable, 1.77.2+): All HTTP, SSE, DB, and credential operations
- Zustand 5: UI state (4 domain stores: agentStore, testStore, workspaceStore, uiStore)
- TanStack Query 5: Async server state for agent card fetches and task polling
- tauri-plugin-sql 2.3.2: SQLite via sqlx — agent cards, history, test cases, workspaces
- tauri-plugin-keyring 0.1.0: OS keychain for API keys and auth headers (unstable — have contingency)
- reqwest + reqwest-eventsource: Rust-side HTTP and SSE stream consumption
- Tauri Channels: SSE chunk delivery from Rust to frontend (ordered, per-task, not global events)
- Monaco Editor (@monaco-editor/react 4.6.x): JSON/text input — must be bundled locally, not CDN
- Tailwind CSS 4.2.x: Utility CSS with Vite plugin — no tailwind.config.js needed

### Expected Features

The feature dependency chain is linear: agent card management is the prerequisite for everything. Without a stored agent card, there are no skills to browse; without skills, there is no adaptive input form; without that, there is no send flow. SSE streaming is not a v2 feature — a large class of A2A agents exclusively use `tasks/sendSubscribe` and cannot be tested without it.

**Must have (table stakes — v1 launch):**
- Agent card management (add by URL, store, display, delete, refresh)
- Skill browser with filtering
- Adaptive skill input panel (text/file/JSON modes based on skill's declared inputModes)
- Send message via tasks/send (synchronous path)
- SSE streaming via tasks/sendSubscribe (async streaming path)
- Task lifecycle display (submitted → working → completed/failed/canceled)
- Response viewer: rendered markdown + JSON tree + raw JSON
- Authentication header support (per-card, keychain-backed)
- Request history (SQLite, searchable)
- Saved test cases with one-click re-run
- Curl export
- Dark/light theme with system detection

**Should have (v1.x — add after core validated):**
- tasks/cancel support
- Named workspaces
- SSE event timeline view
- Artifact viewer (render artifacts by MIME type)
- input-required handling (conversational follow-up on same taskId)
- Agent card diff on refresh
- Per-card settings (base URL override, timeout)

**Defer (v2+):**
- Extended agent card (getExtendedAgentCard) — awaits widespread A2A v0.3 adoption
- Mock agent server — doubles scope; complex protocol server implementation
- Push notification (webhook) testing — requires local server exposure
- Team workspace export/import — evaluate post-PMF

**Deliberate anti-features (keep out of scope permanently):**
- Generic HTTP request builder — would make this Postman-lite and dilute A2A UX
- Agent code editor / IDE features — different product category
- Cloud sync / team workspaces — requires SaaS backend, changes product model
- Test assertions / scripting runtime — scope expansion of several months

### Architecture Approach

The architecture is a three-panel desktop layout (AgentPanel, SkillPanel, TestPanel) communicating through four Zustand stores with all Rust interaction mediated by tauri-specta typed bindings. The critical architectural boundary is the Rust/WebView split: no network calls cross from the webview into the internet. AppState (a Mutex-wrapped struct holding the reqwest Client and active task AbortHandles) is Tauri managed state — all commands share it via `tauri::State<AppState>`. SQLite persists all structured domain data; flat settings use tauri-plugin-store; credentials use the OS keychain exclusively.

**Major components:**
1. React Frontend (WebView): Three-panel UI, Zustand stores, typed command invocations via bindings.ts
2. Tauri IPC Bridge (tauri-specta generated bindings.ts): Type-safe Rust command wrappers; channels for streaming
3. Rust Commands (agents.rs, tasks.rs, history.rs, settings.rs, workspaces.rs): All business logic and I/O
4. A2A Client Module (a2a/client.rs + a2a/sse.rs): Protocol implementation isolated from Tauri machinery
5. AppState: Shared runtime resources — DB pool, HTTP client, active task registry
6. SQLite (tauri-plugin-sql): Five-table schema with WAL mode — agents, workspaces, history, saved_tests, settings

### Critical Pitfalls

1. **Tauri v2 permissions ACL — zero access by default.** The old v1 `allowlist` key is invalid in v2 and hard-errors on build. Every plugin command starts disabled. Establish `src-tauri/capabilities/default.json` on day one with all required permissions (sql:allow-load, sql:allow-execute, sql:allow-select, sql:allow-close, http:default with explicit URL scope). Missing permissions produce cryptic silent failures, not compile errors.

2. **SSE must be implemented in Rust, not the webview.** The browser's `EventSource` API and `fetch()` both hit CORS walls when called from `tauri://localhost`. There is no workaround except moving all HTTP to Rust. This decision must be made and enforced from Phase 1 — retrofitting the HTTP layer after the UI is built is a HIGH-cost recovery.

3. **SQLite requires WAL mode and Rust-side migrations.** Without WAL mode, concurrent async writes cause "database is locked" errors under real usage. Set `PRAGMA journal_mode=WAL` in the first migration. Never use Drizzle's `migrate()` from the webview — it calls `require('fs')` which doesn't exist there. Use tauri-plugin-sql's Rust-side `Migration` structs exclusively.

4. **A2A spec is still evolving.** All agent card parsers must treat non-core fields as optional. Unknown task lifecycle states must not crash the UI. Display raw JSON alongside parsed views so users can debug non-conforming agents.

5. **Monaco Editor requires CSP adjustments and worker bundling.** If Monaco is loaded from CDN or without worker configuration, it silently falls back to single-threaded mode with no console error in production. Use `@monaco-editor/react`, bundle monaco via npm (not CDN), and add `worker-src blob:` + `script-src 'unsafe-eval'` to the Tauri CSP config.

## Implications for Roadmap

The feature dependency chain and pitfall phase-mapping together dictate a clear build order. Foundation must come before any feature work. The A2A client layer must exist before any UI can be built around it. The three-panel UI can then be layered progressively from the outermost (agent management) to the innermost (streaming task execution).

### Phase 1: Foundation and Scaffold

**Rationale:** The Tauri v2 permissions system, SQLite WAL mode, migration strategy, async command patterns, and cross-platform font bundling are all Phase 1 requirements per pitfall analysis. Every subsequent phase builds on these. Getting them wrong means rework throughout the project.

**Delivers:** Working Tauri 2 app scaffold with Rust command bridge, SQLite initialized with WAL mode and full schema, capabilities configured, tauri-specta bindings pipeline, cross-platform font bundling, and Zustand + TanStack Query wired up. No A2A functionality yet, but all infrastructure is correct.

**Addresses:** Dark/light theme (table stakes); application shell and navigation structure

**Avoids:** Permissions ACL misconfiguration (Pitfall 1), SQLite WAL omission (Pitfall 4), migration strategy errors (Pitfall 5), Rust async pattern mistakes (Pitfall 8), cross-platform rendering differences (Pitfall 6)

### Phase 2: Agent Card Management and A2A HTTP Client

**Rationale:** Agent card management is the dependency root for every user-facing feature. The Rust A2A client module (types.rs, client.rs) must be built and tested in isolation before commands expose it. This is also where the Rust-only HTTP pattern is established and validated end-to-end.

**Delivers:** Add agent by URL (fetch /.well-known/agent.json), store in SQLite, display agent card (identity, endpoint, auth schemes, capabilities), skill browser with filtering, agent delete and refresh. Full agent card management feature set from the MVP list.

**Uses:** reqwest (Rust-side HTTP), tauri-plugin-sql, tauri-plugin-keyring (credential scaffolding), tauri-specta generated commands

**Implements:** commands/agents.rs, a2a/types.rs, a2a/client.rs, AgentPanel, SkillPanel, agentStore

**Avoids:** Webview fetch anti-pattern (Pitfall 2), A2A spec defensiveness (Pitfall 10 — optional fields from day one)

### Phase 3: Task Execution — Synchronous Path (tasks/send)

**Rationale:** The synchronous task execution path (tasks/send + poll tasks/get) is simpler than SSE and validates the full request/response lifecycle before adding streaming complexity. Authentication headers, curl export, request history, and saved test cases all attach to this phase because they depend on a working send flow.

**Delivers:** Adaptive skill input panel (text/file/JSON modes), send message via tasks/send, task lifecycle display (state badge with transition history), response viewer (rendered markdown + JSON tree + raw), authentication header support (keychain-backed), request history (SQLite, searchable), saved test cases with one-click re-run, curl export.

**Uses:** Monaco Editor (JSON input), react-markdown + remark-gfm + rehype-highlight (response rendering), tauri-plugin-keyring (auth headers), tauri-plugin-sql (history + saved tests)

**Implements:** commands/tasks.rs (synchronous path), commands/history.rs, TestPanel, InputForm, ResponseViewer, TaskStatus, HistoryList, testStore

**Avoids:** Monaco Web Worker misconfiguration (Pitfall 3 — configure CSP and worker bundling here), IPC large data overhead (Pitfall 9 — paginate history from day one)

### Phase 4: SSE Streaming (tasks/sendSubscribe)

**Rationale:** SSE streaming is table stakes (many A2A agents require it) but depends on the entire send infrastructure from Phase 3 being solid. It is isolated here because it adds a new data flow path (Tauri Channels instead of command return values) and is the most complex feature in the MVP.

**Delivers:** SSE streaming via tasks/sendSubscribe, incremental response display (streaming chunks into ResponseViewer), tasks/cancel support.

**Uses:** reqwest-eventsource (Rust), Tauri Channels (ordered streaming), testStore.appendChunk, AbortHandle in AppState

**Implements:** a2a/sse.rs, commands/tasks.rs stream_task command, useStreamingTask hook, Channel<TaskEvent> pipeline

**Avoids:** SSE in webview anti-pattern (Pitfall 2 — enforced by architecture), SSE byte-level emit overhead (buffer on SSE event boundaries, not raw bytes)

### Phase 5: Enhanced Experience (v1.x features)

**Rationale:** Once the core testing workflow is validated with real users, add the features that address the most common power-user friction: workspace isolation, visual streaming event timeline, artifact rendering, input-required conversational handling, and per-card configuration.

**Delivers:** Named workspaces (scope agents/history/tests per environment), SSE event timeline view (append-only ordered event log), artifact viewer (MIME-type aware rendering), input-required handling (detect state, prompt for follow-up, send on same taskId), agent card diff on refresh, per-card settings (base URL override, timeout).

**Research flag:** input-required handling requires careful understanding of A2A task continuation semantics. Phase-level research recommended before implementation.

### Phase 6: Distribution Preparation

**Rationale:** macOS notarization, Windows code signing, and installer packaging require a paid Apple Developer account and separate CI configuration. These are not code features — they are build pipeline concerns that must be addressed before any external beta distribution.

**Delivers:** Signed and notarized macOS .dmg, signed Windows .exe installer, Linux AppImage. CI pipeline for automated builds.

**Avoids:** macOS "app is damaged" Gatekeeper block (Pitfall 7), Windows SmartScreen rejection

### Phase Ordering Rationale

- Phase 1 before everything: The permissions ACL, WAL mode, and async command patterns are load-bearing. Every other phase inherits these foundations.
- Phase 2 before Phase 3: The feature dependency graph is strict — skills require agent cards; the send form requires skills.
- Phase 3 before Phase 4: SSE streaming is an alternate transport for the same message flow; the request construction, auth, history, and response display infrastructure must exist first.
- Phase 5 after user validation: Named workspaces, SSE timeline, and input-required handling add complexity. Add them when usage patterns confirm the need.
- Phase 6 independent: Code signing is a build concern, not a code concern. Budget for it early but execute it at the distribution milestone.

### Research Flags

Phases needing deeper research during planning:
- **Phase 4 (SSE / input-required):** input-required task continuation semantics in A2A spec are complex; verify against current spec before designing the state machine
- **Phase 6 (Distribution):** macOS notarization CI pipeline and Windows signing certificates change frequently; verify current Tauri 2 recommendations at implementation time

Phases with standard, well-documented patterns (research-phase can be skipped):
- **Phase 1:** Tauri 2 scaffold is fully documented; pitfalls are known and documented in PITFALLS.md
- **Phase 2:** reqwest + tauri-plugin-sql patterns are established; agent card parsing is straightforward JSON
- **Phase 3:** Monaco + react-markdown + TanStack Query patterns are standard; pitfalls documented

### Gaps to Address

- **tauri-plugin-keyring stability:** v0.1.0 is community-maintained and marked unstable. Validate credential read/write on macOS, Windows, and Linux in Phase 1. If unreliable, implement the fallback (AES-encrypted SQLite using a machine-specific key) before Phase 3 introduces real credentials.
- **A2A spec version targeting:** The spec is evolving. Pin to a specific spec version at project start and document which version the tool targets. Revisit when getExtendedAgentCard (v0.3) is implemented.
- **Monaco bundle size:** Monaco adds ~4MB to the frontend bundle. Lazy-load the editor only when the test panel is first opened. Validate that cold-start time stays under 2 seconds.
- **Concurrent task support:** Current design supports one active task at a time (one AbortHandle per task concept). If users request parallel task execution, the active_tasks HashMap can expand, but the UI will need rework.
